Continuous Integration

Continuous Integration is a software development practice where members of a team integrate their work frequently.Each integration is verified by an automated build (including test) to detect integration errors as quickly as possible.Many teams find that this approach leads to significantly reduced integration problems and allows a team to develop cohesive software more rapidly. [1]

Martin Fowler

TIPS to build your first Continuous Integration Project
  • Get everything you need into source control get it so that you can build the whole system with a single command.
  • Introduce some automated testing into your build
  • Try to identify the major areas where things go wrong and get automated tests to expose those failures
  • Try to speed up the commit build.
  • Every Commit Should Build the Mainline on an Integration Machine
  • Fix Broken Builds Immediately
  • Test in a Clone of the Production Environment
  • Make it Easy for Anyone to Get the Latest Executable
  • Automate Deployment
Continuous Delivery

Continuous Delivery is a software development discipline where you build software in such a way that the software can be released to production at any time.You achieve continuous delivery by continuously integrating the software done by the development team, building executables , and running automated tests on those executables to detect problems. [2]

Martin Fowler

TIPS for continuous delivery across your project :

  • DevOpsCulture:A close, collaborative working relationship between everyone involved in delivery.
  • DeploymentPipeline:Extensive automation of all possible parts of the delivery process
How my Project is setup with GitHub Actions:
GITHub Actions is the CI/CD pipeline we are using for our project.
  • 1. Create a new GitHub repository
  • 2. Create a new Docker repository in Dockerhub
  • 3. Create Heroku production app
  • 4. Create Heroku Development app
  • 5. Create API key within Heroku
  • 6. Add “secrets” to the GitHub
  • 7. Update info in prod.yml and dev.yml workflow files
Development Workflow

Whenever a Pull Request is created for changes, the development workflow is triggered. During the development workflow, the following events happen:

  • Runs test for configuration,context process and html pages.
  • The image is then deployed to the dev app on Heroku only if all test cases pass.
  • Any new commits that go into the PR will trigger the workflow again.
  • feature changes can be viewed on the dev heroku app's url.
  • Using the dev app's url anybody can review the code.
Production Workflow

When the Pull Request is merged into master, the production workflow is triggered. In the production workflow, the following events happen:

  • changes are committed from dev to master.
  • The merge triggers the workflow .
  • Runs test for configuration,context process and html pages.
  • The docker image is built and uploaded to the docker hub.
  • The docker image is then deployed to the dev app on Heroku
  • Using the prod app's url you can check the workflow.
