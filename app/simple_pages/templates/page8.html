SOLID Demonstration in Calculator App

S - Single-responsiblity Principle

A class should have one and only one reason to change, meaning that a class should have only one job.

In our calculator app , this principle is demonstrated as follows:

In Calculation.py we have individual classes that perform only one job. For eg. Multiplication class will only one job multiply the values provided

O - Open-closed Principle

Objects or entities should be open for extension but closed for modification.This means that a class should be extendable without modifying the class itself.

In our calculator app , this principle is demonstrated as follows:

when our child class Addition extends the parent class ,the methods and attributes of parent class are inherited in Addition class. This methods can be extended in any number of child classes but it does not modify the actual parent class. If we want to add a new operation like square root we simply can extend a new class called square root 


          
L - Liskov Substitution Principle

            
every subclass or derived class should be substitutable for their base or parent class. 

            
In  our calculator app , this principle is demonstrated as follows: 

            
Inside the calculation class ,we extend the addition,subtraction,multiplication and division class.
                Changing or removing this child class does not affect our parent class in any way

          

        

                    
                    
        


        

            
            

            
I - Interface Segregation Principle

            
A client should never be forced to implement an interface that it doesn’t use, or clients
                shouldn’t be forced to depend on methods they do not use.

            
In  our calculator app , this principle is demonstrated as follows: 

            
Inside the calculation class ,the subclass multiplication only uses the
                multiply function from operation class.It doesn't use any unnecessary operation or calculation functions
                not needed.Similarly, it goes for every other classes Addition,Subtraction,Division.
            



        

        

                    
                    

        

        

            
            

            
D - Dependency Inversion Principle

            
Entities must depend on abstractions, not on concretions. It states that the high-level module must not
                depend on the low-level module, but they should depend on abstractions. 

            
In  our calculator app , this principle is demonstrated as follows: 

            
Inside the main calculator class ,we call the multiplication class from calculations class.
                Inside the multiplication class ,we call the get result method.Inside the get result method,the multiply
                method from operations is called. Finally only output of get result method is retrieved from  the main
                calculator class

        

        

        
Design Pattern 

            
                
Design patterns are typical solutions to common problems in software design. Each pattern is like
                    a blueprint that you can customize to solve a particular design problem in your code.

                
Types of patterns:

                
    
                 
  1. Creational patterns   provide object creation mechanisms that increase flexibility and reuse of existing code.
    2. 
                    
  2. Structural patterns explain how to assemble objects and classes into larger structures, while keeping these structures
                        flexible and efficient.
    3. 
                    
  3. Behavioral patterns take care of effective communication and the assignment of responsibilities between objects.
    4. 
                

            
        

        

         
Demonstration of  creational pattern called Factory Method in Calculator App :

          
  Factory Method is a creational design pattern that provides an interface for creating objects in a superclass,
            but allows subclasses to alter the type of objects that will be created

            
In our calculation class ,we have a factory method to create a tuple_list of input values,but the type of
                object list which will be created  is decided in subclasses addition ,multiplication and so on
